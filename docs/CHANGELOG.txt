NeoDCT Changelog

Version 0.3.0a (in progress) - 2026-07-14

Modem (SIM7600G bring-up -- MODEM SUPPORT!)
- QEMU kernel config now ships the drivers the SIM7600 needs and they are
  persisted in board/qemu/aarch64-virt/linux.config (they used to live only
  in the ephemeral build .config): USB_SERIAL_OPTION (the driver that
  actually creates /dev/ttyUSB0-4 for 1e0e:9001 -- its absence was why
  lsusb saw the modem but no tty nodes appeared), plus QMI_WWAN for a
  wwan0 interface at the stock PID, ACM/usbnet/RNDIS carried over.
- New atcmd helper (engineering/tools/atcmd): one scriptable AT command
  per call over BusyBox sh, with an advisory lock shared with the boot
  script and the UI so replies never interleave.
- New /etc/init.d/S45modem: backgrounded boot bring-up -- SIM check, PDP
  context (IPV6 / fast.t-mobile.com by default, /etc/default/modem to
  override), LTE registration wait, AT$QCRMCALL data call, wwan0 + SLAAC
  + DNS64. Logs to /tmp/modem-boot.log, status in /tmp/modem.status.
- ModemService is REAL now: owns the AT port via raw termios (no pyserial,
  no ModemManager, no threads), auto-probes ports by USB interface number,
  polls AT+CSQ/+CEREG/+COPS on staggered timers from the main loop, parses
  call URCs (RING/+CLIP/VOICE CALL/NO CARRIER) into a state machine plus
  event queue, and dial()/hangup() send real ATD/AT+CHUP. Falls back to
  Simulation Mode like BatteryService (sim hooks: /tmp/neodct_sim_csq,
  /tmp/neodct_sim_ring) and hot-adopts a modem that appears later.
- Home screen signal bars are LIVE: the sig icon_set now renders 0-4 bars
  from real tower CSQ (0 bars when unregistered); sim_val still applies in
  Simulation Mode.
- Home screen carrier line is LIVE: the "No Service" text shows the real
  operator name ("Tello"/"T-Mobile", AT+COPS short format) while
  registered, and reverts to No Service when registration drops.
- Call placement is safety-gated while the voice path is unfinished:
  dial()/answer() only send real ATD/ATA when the new
  system.modem.allow_calls setting is ON (default OFF -> pretend flow,
  2 s fake connect, even on live hardware). Hangup still sends AT+CHUP
  (harmless when idle; rejects a real incoming ring).
- docs/MODEM_BRINGUP.md: full runbook -- QEMU passthrough gotchas (host
  ModemManager, PID-switch warning), terminal proof-of-concept sequence,
  S45modem config, ModemService integration notes, troubleshooting table.
- New Modem engineering app (id 9005, signal-bars placeholder icon):
  three status pages walked with the softkey (Next/Next/Exit, Back exits
  anywhere) in the FuelGauge readout style. RADIO: operator, CEREG state,
  CSQ + dBm, bars, call state (1 s refresh). SIM: CPIN, phone number
  (+CNUM), IMEI, ICCID (+CICCID with +CCID fallback), IMSI, firmware --
  queried once per visit since identity never changes. DATA: S45modem
  boot status, wwan interface + up/down, global IPv6, APN, DNS. Unlike
  FuelGauge it does NOT exit in Simulation Mode: it shows which ttyUSB
  nodes exist instead, so "why is there no modem" is debuggable on-screen
  without a serial console.

Calls -- INCOMING (the phone rings!)
- The phone now rings on incoming calls and interrupts whatever app is
  running, like the real 3310 (it didn't multitask either; a nicer
  suspend/resume is post-alpha work). Mechanism: ModemService's RING/
  +CLIP URCs are seen by read_keypress -- the chokepoint every screen
  funnels through -- which raises IncomingCall. It derives from
  BaseException on purpose (exactly like KeyboardInterrupt) so an app's
  own `except Exception` can't swallow a ringing phone, and unwinding
  runs the apps' finally blocks: MusicPlayer/Koki release the ALSA
  device before the ringtone starts.
- New incoming-call screen (System/ui/Dialer/incoming_screen.py) built
  to the 3310 photo: caller name (phonebook lookup by last-10-digits, so
  a stored "555-1234" matches a "+1555..." caller ID) or number across
  the top, auto-shrunk so a full USA number never runs off screen;
  flashing "calling" near the bottom left; softkey says Answer and C
  declines. A late +CLIP updates the name live instead of leaving
  "Unknown". Caller hangs up first -> the screen exits by itself.
- Answer -> AT+ATA -> the existing in-call screen with USB call audio;
  decline/hangup -> AT+CHUP. The ringer always stops before call audio
  claims the ALSA device.
- NotifyService grew the ringer: the user's ringtone
  (system.audio.ringtone) is decoded once and looped in-process with
  python-miniaudio (same reasoning as MusicPlayer: no ~24MB mpv
  process), mpv only as fallback. Paths with spaces are handled
  natively -- the path never touches a shell.
- Ringtone resolution never rings silently: a missing file falls back to
  the same basename with another extension (so the .wma -> .mp3
  re-encode keeps working), then Nokia Tune/Ring Ring, then any playable
  tone, each logged. NOTE: settings.prop currently points at "Brave
  Scottland.mp3" but the file is "Brave Scotland.mp3" (one t), so it
  rings the fallback until the typo is fixed.
- SettingsStorage default ringtone updated .wma -> .mp3 (the shipped
  tones were re-encoded).
- Sim hook fixed: /tmp/neodct_sim_ring now rings once per write/touch
  (mtime-tracked) instead of instantly re-ringing after a decline, which
  made declining impossible to test; `rm` while ringing still means the
  caller gave up.
- Verified against the real main.py code paths: IncomingCall dodging an
  app's `except Exception` while its finally still runs, _ring_tick
  grace period for +CLIP (and withheld numbers), the answer/decline/gone
  flows with ring-stop ordering, ringtone resolution (typo'd path, space
  in name, .wma->.mp3, empty setting), the loop generator's seamless
  wrap, and the sim ring hook.

Calls (REAL VOICE CALLS -- receive audio only for now)
- Dialing is live: system.modem.allow_calls now defaults ON (set OFF for
  the old pretend flow). dial() sends ATD, then immediately AT+CPCMREG=1
  to route call audio over USB, then pipes the modem's PCM port (USB
  interface 4, auto-detected like the AT port; 16 kHz mono S16_LE) to
  the speaker via `aplay -t raw` -- pure BusyBox plumbing, no in-process
  audio code, kilobytes of overhead. Receive-only by design: there is no
  microphone hardware yet, so the far end hears silence. If the network
  sends early media you hear real ringback while it rings (local fake
  ringback is a future fallback).
- Call teardown from every direction kills the audio pipe and schedules
  AT+CPCMREG=0: End key (AT+CHUP), remote hangup (NO CARRIER /
  VOICE CALL: END), missed. The in-call screen now shows "Calling..."
  until connected and exits on its own when the far end hangs up
  (previously it sat there until you pressed End).
- Fake DTMF dial beeps while typing a number on the home screen: twelve
  proper dual-tone wavs (0-9 * #, 697-1477 Hz pairs, 120 ms, click-free
  ramps) generated into System/tones/dtmf/. VoLTE never hears them --
  they just sound right.
- Verified against the pty fake modem: ATD -> CPCMREG=1 ordering, remote
  -end teardown with deferred CPCMREG=0, hangup path, allow_calls=OFF
  pretend flow, and wav format checks. First live test target:
  1-800-444-4444 (MCI's readback line).
- Call monitoring after the first live test showed calls connecting but
  no way to see what the audio path did: the console now logs every
  call URC, the CPCMFRM/CPCMREG results at dial time (CPCMREG=1 is
  re-asserted after connect for firmwares that reject it early), and
  AT+CLCC is polled every 2 s during a call -- Call progress:
  CALLING -> RINGING -> CONNECTED transitions in the log, "Ringing..."
  on the call screen, and a running mm:ss timer once connected. An
  empty CLCC list while a call is supposedly up ends the call state
  (recovers missed VOICE CALL: END). An aplay watchdog logs the exit
  code and restarts the pipe if it dies mid-call. AT+CPCMFRM now
  explicitly selects 16 kHz (factory default is 8 kHz on many builds --
  a prime silent-audio suspect); system.hw.modem_pcm_rate=8000 flips
  both the modem and aplay if a firmware insists on 8 k.

Messages (SMS sending -- FIRST LIVE TELEPHONY!)
- The Messages app really sends SMS now: Write Message -> Options ->
  Send prompts a "Number" TextInput, then hands off to the modem and
  reports "Message sent!" (with the network's +CMGS reference) or the
  exact failure. The old "requires Telephony... by M3" placeholder is
  gone from both the composer and the Outbox draft Send option.
- Pressing any arrow key on the Number prompt opens the shared PhoneBook
  contact selector (first real use of that groundwork) and autofills the
  picked contact's number.
- ModemService.send_sms(): text-mode AT+CMGS with bespoke handling for
  the ">" prompt (it arrives without a newline, invisible to the line
  reader), 30 s network-ack window, ESC bailout if the prompt never
  comes, CMS/CME error reporting, and ascii-safe body encoding. Verified
  end-to-end against a pty-based fake modem speaking the real CMGS
  dance, plus rejection/validation paths. Simulation Mode pretends
  honestly ("simulated").
- Guardrails: empty drafts and >160-char messages are stopped in the app
  with a clear dialog before the modem is ever touched.
- RECEIVING is live too, 3310-style (new NotifyService,
  System/core/NotifyService): a new SMS beeps the original Nokia tone
  immediately (System/tones/sms.wav, aplay fire-and-forget; invisible to
  the ringtone list which only scans mp3/wma) and puts an "N message(s)
  received" banner on the HOME screen -- carrier line steps aside,
  softkey becomes Read (straight into the message, or the Inbox when
  several arrived), C dismisses. A flashing envelope sits in the status
  strip while unread mail exists. Nothing pops over running apps (by
  design, like battery warnings); battery keeps its own framework.
- Push, not polled: AT+CNMI raises +CMTI the instant a message lands ->
  ModemService.fetch_sms (AT+CMGR) -> existing sms_inbox.db -> notify.
  A boot sweep (AT+CMGL="REC UNREAD") imports messages that arrived
  while the phone was off. Fetched messages are deleted from the SIM so
  its ~30 slots can't fill up and silently kill reception. If the AT
  port is busy (S45modem still registering) the event politely requeues.
- Opening a message finally marks it read (is_read was never updated
  before -- inbox stars never cleared). Sim hook for QEMU:
  echo '5551234|hey' > /tmp/neodct_sim_sms.
- Verified against the pty fake modem: CMGL boot sweep (multi-line
  bodies), CMTI->CMGR->CMGD flow, busy requeue, sim hook, banner
  pluralization/dismiss.

MusicPlayer
- Playback engine swapped from spawned mpv processes to in-process
  python-miniaudio streaming (same library as Koki's mixer): tracks
  decode chunk-by-chunk straight into one ALSA device, so playing a song
  costs kilobytes of buffer instead of a ~24MB mpv process (the usual
  OOM trigger on real hardware). mpv remains an automatic fallback when
  miniaudio is missing or its device fails (NEODCT_MUSIC_AUDIO=
  subprocess forces mpv; =miniaudio disables the fallback).
- flac/ogg now playable; aac only via the mpv fallback (miniaudio has
  no aac decoder).
- Progress bar/timestamps track the real decoded position under
  miniaudio instead of the wall-clock estimate (which stays for mpv).
- Track duration now reported for wav/flac/ogg too via miniaudio
  get_file_info (mutagen only handled mp3).
- BR2_PACKAGE_PYTHON_MINIAUDIO=y added to the qemu + luckfox defconfigs
  (it lived only in the live .config; a clean rebuild would drop it).

Version 0.2.4a - 2026-07-13

Power/Battery
- New BatteryService (System/core/BatteryService): reads VCELL from a
  MAX1704x fuel gauge over I2C (bus/addr configurable via
  system.hw.battery_i2c_bus / system.hw.battery_i2c_addr, default 1/0x36).
- Home screen battery gauge is now live: <3.35 V dead (bat-0), 0.20 V bands
  up to >=3.95 V full (bat-4). Readings are smoothed over a 5-sample window.
- LOW BATTERY! warning at <=3.45 V, BATTERY CRITICALLY LOW! at <=3.25 V
  (one-shot, re-armed if the pack recovers), graceful poweroff at <=3.20 V
  (confirmed over 3 consecutive polls; syncs filesystems first).
- QEMU/dev fallback: without gauge hardware the service runs in Simulation
  Mode (boot log says which mode is active, like the modem stub). Simulated
  voltage overridable via NEODCT_BATT_SIM_VCELL env var or by writing a
  value to /tmp/neodct_sim_vcell at runtime.
- Added engineering/tools/max1704x_watch.py bench utility used for the
  Raspberry Pi supply-sweep prototyping.

Engineering
- New FuelGauge engineering app (id 9004, diesel-can icon): live MAX1704x
  readout (VCELL/SOC/CRATE/VERSION/CONFIG + system gauge state) refreshed
  every second through the shared BatteryService handle. Softkey issues a
  gauge quick-start (re-seeds SOC from VCELL for bench sweeps). Shows a
  notice and exits when the gauge is simulated (QEMU / missing hardware),
  like the other hardware-bound engineering apps.
- BatteryService grew debug_snapshot() and quickstart() to back the app.
- Engineering apps finally lost the hard-hat placeholder icon: CubeBench
  (3D cube), KeypadMapper (access-keypad photo), KeypadMapperI2C (same
  photo + I2C badge), Crash (red X ball, redrawn crisp), LCDTest
  (monitor) and Tests (checklist clipboard). All 120x120 transparent
  PNGs; FuelGauge and LinuxShell keep their existing icons.

Framework
- MessageDialog.render() draws a dialog without blocking on a key;
  ErrorScreen.show_error(wait_for_ack=False) exposes it for the shutdown
  notice.
- MessageDialog layout fixes: body text no longer overlaps the warning
  icon (affected every dialog incl. the alpha notice), the body block is
  vertically centered, and short notices (<=2 lines) now render
  Nokia-alert style: normal font, centered. Paragraphs keep the small
  left-aligned layout.
- Home layout icon_set elements now honor sim_val instead of a hardcoded
  level (cell signal still simulated until the modem goes real).

Versioning
- Version bumped to 0.2.4a; stale version strings in an existing
  settings.prop are now refreshed to the installed build.

Version 0.2.0a - 2026-02-25

Summary
- Major milestone release focused on real hardware constraints and performance.
- Moved directly to 0.2.0a due to scope and platform changes.

Build Organization and Milestones
- 0.2 introduces a cleaner split between software versioning and hardware progress tracking.
- Build identifiers now stay software-only (for example: `v0.2.0a`), without milestone tags in the build name.
- Hardware stages (PROTO/EVT/DVT/PVT) are now treated as parallel project-tracking states, not version suffixes.
- This allows simultaneous work across tracks (for example, PROTO0 experiments and EVT0 target bring-up in the same timeframe) without version-label confusion.
- Legacy milestone-style build labels were dropped as part of this refactor.

Platform
- Primary SBC target changed from Radxa Zero 3W to Luckfox Pico Mini B.
- Luckfox profile now reflects practical embedded constraints for NeoDCT engineering:
  - CPU: single Cortex-A7 @ 1.2 GHz
  - RAM: 64 MB
  - Storage: built-in 128 MB SPI flash + SD expansion
- Buildroot is now running on real hardware for this track.

System/Buildroot
- Slimmed package set for core UI path and removed heavyweight desktop/browser stack from the lean configs:
  - removed/disabled: Cage, Wayland/Wlroots/Wlr-randr, X.org stack, WebKitGTK, NetSurf, Mesa stack, and related extras in lean profiles.
- Kept minimum runtime set for NeoDCT UI and lightweight browsing workflow:
  - Python3 (+ssl/sqlite/xz), Pillow, FreeType, DejaVu, CA certs, Links (+graphics), GPM.
- Re-added python-mutagen for Music Player metadata/art handling.
- Fixed overlay pathing in active configs to `../neodct/overlay`.
- Added split-output workflow support (`O=...`) for separate QEMU and Luckfox build trees.

Rendering/Performance
- Core framebuffer pipeline in `System/core/main.py` was optimized (no external renderer split):
  - reduced per-frame allocations by reusing buffers/images
  - reduced unnecessary conversions/crops
  - optimized RGB565 fallback path with lookup tables and reusable output buffers
  - added partial center-band write path for common UI case to avoid unnecessary full-frame work
- Result: noticeably smoother software rendering in QEMU bottleneck testing and improved perceived UI smoothness.

Engineering/Apps
- Engineering app layout standardized under `/NeoDCT/System/engineering`.
- Added CubeBench engineering app (pure software PIL 3D spinning cube) as a baseline rendering benchmark.
- Added Luckfox-specific init behavior:
  - `inittab.luckfox` with serial console handling for hardware bring-up
  - guarded post-build application so Luckfox-specific inittab does not leak into QEMU builds
  - serial redundancy configured for `ttyFIQ0` and `ttyAMA0` in Luckfox profile
- Luckfox post-build currently prunes `/tests` for smaller rootfs in that profile.

Versioning/UI
- Updated version strings/banners to 0.2.0a context.
- QEMU banner and settings fields were aligned during transition work.

Browser Direction
- WebKitGTK is no longer viable for the 64 MB RAM target.
- Browser app path has shifted to lightweight Links for now.
- Full web experience will require a reworked approach suitable for ultra-constrained memory.

Notes
- This release is primarily an under-the-hood engineering leap toward true device-level constraints, not just emulator convenience.
- The new constraints are intentionally closer to classic feature-phone-era engineering tradeoffs.
