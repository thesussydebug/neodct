MusicPlayer: swap mpv subprocess for in-process miniaudio streaming

Playing a track no longer spawns mpv (~24MB private RSS, the OOM
killer's first pick on the 64MB targets). The new _MiniaudioPlayer
streams the file chunk-by-chunk through python-miniaudio into one
ALSA device -- kilobytes of buffer, no spawn latency, and a single
ALSA client (same reasoning as Koki's mixer).

- pause/resume = device stop/start with the same stream generator
  (it keeps its position), replacing SIGSTOP/SIGCONT on mpv
- progress bar shows the real decoded-frame position; the wall-clock
  estimate remains only for the mpv fallback
- end-of-track via stream_with_callbacks end_callback
- stream decode buffer sized to the device period (default 500ms,
  NEODCT_MUSIC_ABUF_MS overrides): miniaudio raises mid-callback if
  the device asks for more frames than frames_to_read allocated --
  found by the headless test; the track would hang silent forever
- mpv kept as automatic fallback when miniaudio is missing or its
  device fails to init; a bad file (DecodeError) does NOT demote the
  session to mpv. NEODCT_MUSIC_AUDIO=subprocess|miniaudio forces a
  backend (same convention as NEODCT_KOKI_AUDIO)
- flac/ogg now scanned + playable; aac only via the mpv fallback
  (miniaudio has no aac decoder)
- wav/flac/ogg durations via miniaudio.get_file_info (mutagen only
  handled mp3)
- BR2_PACKAGE_PYTHON_MINIAUDIO=y added to qemu + luckfox defconfigs
  (it lived only in the live .config; a clean rebuild would drop it)

Verified headlessly (NULL backend, real app module, UI stubbed):
play/pause/resume/position, end detection, replay after stop,
bad-file and device-failure fallback paths, plus a real mp3 decode
sweep -- 17 checks + mp3, all green. Also noted in the 0.3.0a
CHANGELOG MusicPlayer section.

INCOMING CALLS: the phone rings, interrupts apps, Answer/decline

Round one of telephony is complete -- NeoDCT now receives calls.

- IncomingCall(BaseException) raised from read_keypress (the chokepoint
  every screen funnels through) unwinds whatever app is running. It
  derives from BaseException exactly like KeyboardInterrupt so an app's
  `except Exception` can't swallow a ringing phone; unwinding still runs
  the apps' finally blocks, so MusicPlayer/Koki release the ALSA device
  before the ringtone starts. launch_app/render_menu re-raise it instead
  of showing the crash screen; the core loop catches it and rings. The
  3310 didn't multitask either -- suspend/resume is post-alpha.
- New System/ui/Dialer/incoming_screen.py, built to the 3310 photo:
  caller name or number across the top (auto-shrunk; a full USA number
  never runs off), flashing "calling" bottom-left, softkey "Answer",
  C declines. Phonebook lookup matches on the last 10 digits so a stored
  "555-1234" resolves a "+1616555..." caller ID. A late +CLIP updates
  the label live; _ring_tick waits one poll cycle for it so the screen
  doesn't open on "Unknown" (withheld numbers still ring after 0.6 s).
  Caller hangs up first -> screen returns "gone" on its own.
- NotifyService gained the ringer: system.audio.ringtone decoded once
  and looped in-process via python-miniaudio (no ~24MB mpv), mpv as
  fallback. Paths with spaces never touch a shell. Resolution chain
  never rings silently: exact path -> same stem with another extension
  (survives the .wma->.mp3 re-encode) -> Nokia Tune/Ring Ring -> any
  playable tone, each step logged.
  NOTE: settings.prop says "Brave Scottland.mp3"; the file is "Brave
  Scotland.mp3" (one t) -- rings the fallback until that's fixed.
- SettingsStorage default ringtone .wma -> .mp3.
- Sim hook fix: /tmp/neodct_sim_ring rings once per write/touch
  (mtime-tracked). It used to re-ring instantly after a decline, making
  the decline path untestable; `rm` while ringing still = caller gave up.
- Ring volume control in Tones is deliberately left for later.

Tested against the real main.py/NotifyService/ModemService code (PIL and
UI stubbed): exception semantics through a greedy app handler + finally,
_ring_tick CLIP grace and withheld-number path, answer/decline/gone
flows asserting ring-stop happens before answer takes ALSA, ringtone
resolution (typo, space, wma->mp3, unset), loop generator wrap, and the
sim ring hook lifecycle.

FULL-DUPLEX CALLS: mic uplink via arecord into the PCM port

The other half of the audio path -- a complete NeoDCT OS phone call.
The SIM7600's CPCMREG stream is bidirectional, so the uplink mirrors
the downlink with pure alsa-utils plumbing:

    speaker <- aplay   <- /dev/ttyUSBn   (far end's voice, existing)
    far end <- /dev/ttyUSBn <- arecord   (USB sound card mic, NEW)

- arecord -q -t raw -f S16_LE -r <pcm_rate> -c 1 -D <dev> <port>,
  spawned next to aplay at dial/answer, killed with it at teardown.
  arecord ships in the image already (alsa-utils APLAY installs both).
- system.hw.modem_mic_device picks the capture device ("default";
  "plughw:N,0" to pin; OFF/NONE for deliberate listen-only).
- Watchdogs split: speaker restarts indefinitely (3 s holdoff), the mic
  gets three attempts then gives up into listen-only with a logged hint
  (`arecord -l`) -- a guest without a capture device is a normal state,
  not something to spin on.
- Real hardware uses an embedded stripped USB sound card = the exact
  same snd-usb-audio path as QEMU passthrough; this plumbing is final.
  If QEMU's usb-audio turns out playback-only (`arecord -l` empty),
  virtio-sound-pci does capture and CONFIG_SND_VIRTIO=y already ships.

Tested with recorded-Popen fakes over a pty PCM port: dual spawn with
correct args, mic three-strikes -> listen-only (then stays quiet),
speaker restart, hangup kills all live pipes, OFF/custom device
settings pass through. py_compile clean.

mic uplink: auto-detect the capture card (QEMU usb-audio is playback-only)

First full-duplex field test confirmed QEMU's usb-audio device exposes
zero capture endpoints (arecord -l empty in the guest) -- the mic pipe
did its 3-strikes give-up as designed. The fix is passing the USB mic
itself through (same snd-usb-audio path as the real phone's embedded
sound card), which creates a second guest sound card... where ALSA's
"default" (card 0 = playback-only QEMU USB Audio) becomes a trap.

- system.hw.modem_mic_device now defaults to AUTO: scan /proc/asound
  for the first cardN with a pcm*c (capture) node -> plughw:N,M.
  Explicit devices still pass through; OFF still means listen-only.
- No capture device found = one clean log line and a listen-only call,
  not three arecord deaths.
- Also confirmed in the field: CPCMREG=1 -> ERROR at dial time on this
  firmware, then OK on the post-connect retry -- the retry path earns
  its keep.

Tested against a fake /proc/asound tree: playback-only card 0 + mic on
card 1 -> plughw:1,0; capture on card 0 (real-hardware shape) ->
plughw:0,0; no capture -> mic pipe skipped; hotplug found on next call.
